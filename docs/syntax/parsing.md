> Таблица стилей Sass анализируется из последовательности кодовых точек Unicode. Она анализируется напрямую, без предварительного преобразования в поток токенов.

## Входная кодировка {#input-encoding}

!!! note "Примечание"

    Dart Sass в настоящее время поддерживает *только* кодировку UTF-8. Поэтому безопаснее всего кодировать все таблицы стилей Sass в UTF-8.

Часто документ изначально доступен только как последовательность байтов, которая должна быть декодирована в Unicode. Sass выполняет это декодирование следующим образом:

* Если последовательность байтов начинается с кодировки UTF-8 или UTF-16 символа U+FEFF BYTE ORDER MARK, используется соответствующая кодировка.

* Если последовательность байтов начинается с простой ASCII-строки `@charset`, Sass определяет кодировку, используя шаг 2 алгоритма CSS для [определения резервной кодировки](https://drafts.csswg.org/css-syntax-3/#input-byte-stream).

* В противном случае используется UTF-8.

## Ошибки парсинга {#parse-errors}

Когда Sass обнаруживает недопустимый синтаксис в таблице стилей, парсинг завершается с ошибкой, и пользователю выводится сообщение с информацией о местоположении недопустимого синтаксиса и причине ошибки.

Обратите внимание, что это отличается от CSS, который определяет, как восстанавливаться после большинства ошибок, а не немедленно завершаться. Это один из немногих случаев, когда SCSS не является *строго* надмножеством CSS. Однако для пользователей Sass гораздо полезнее видеть ошибки немедленно, а не передавать их в вывод CSS.

Доступ к местоположению ошибок парсинга можно получить через специфичные для реализации API. Например, в Dart Sass вы можете получить доступ к [`SassException.span`](https://pub.dartlang.org/documentation/sass/latest/sass/SassException/span.html), а в JS API Node Sass и Dart Sass вы можете получить доступ к свойствам [`file`, `line` и `column`](https://github.com/sass/node-sass#error-object).
